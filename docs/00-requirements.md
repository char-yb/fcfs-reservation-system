# Requirements

> 한정 수량 숙소 상품을 대상으로 하는 예약·결제 시스템의 구현 요구사항을 정리한다.
> 공개 저장소에 포함될 수 있도록 특정 회사명과 외부 안내 맥락을 제외하고, 시스템 요구사항 중심으로 재작성했다.

---

## 1. 서비스 시나리오

- 특정 시간인 00시에 오픈되는 초특가 숙소 상품을 대상으로 한다.
- 상품 수량은 10개 한정이다.
- 2대 이상의 애플리케이션 서버로 구성된 분산 환경을 가정한다.
- 모든 사용자에게 동등한 구매 기회를 제공해야 한다.
- 재고 정합성을 엄격하게 유지해야 한다.
- 결제 실패와 시스템 장애 상황에서도 안정적으로 동작해야 한다.

---

## 2. 기술 요구사항

| 구분 | 요구사항 |
|---|---|
| Language | Java 8 이상 또는 JVM 기반 언어 |
| Framework | Spring Boot 2.7 이상 |
| RDB | MySQL 또는 MariaDB 계열 |
| Cache | Redis |
| Infra | 애플리케이션 서버 2대 이상의 분산 환경 가정 |
| Other | 필요한 기술은 자유롭게 선택하되 도입 근거를 `DECISIONS.md`에 기록 |

---

## 3. 필수 API

### 3.1 Checkout API

주문서 진입 API다. 다음 정보를 조회할 수 있어야 한다.

- 상품 정보
- 상품 가격
- 입실/퇴실 시간
- 사용자의 가용 포인트

### 3.2 Booking API

결제와 예약 완료를 처리하는 API다.

- 주문서 정보를 입력받는다.
- 결제를 진행한다.
- 최종 주문을 생성한다.
- 요청/응답 규격은 시스템 구조에 맞게 설계할 수 있다.

---

## 4. 핵심 요구사항

### 4.1 재고 정합성 및 공정성

- 00시 트래픽 집중 상황에서 초과 판매가 발생하지 않아야 한다.
- 실제 재고보다 적게 판매되는 상황도 최소화해야 한다.
- 모든 사용자가 동등한 확률로 상품을 구매할 수 있는 구조를 고민해야 한다.

### 4.2 고가용성

- TPS 급증 상황에서도 시스템 붕괴를 막을 수 있는 구조를 반영해야 한다.
- 평시 트래픽은 약 50 TPS로 가정한다.
- 프로모션 시작 직후 1~5분 동안 500~1000 TPS 피크를 가정한다.
- 피크 트래픽 대응 구조와 근거를 `DECISIONS.md`에 기록해야 한다.

### 4.3 멱등성

- 주문서에서 짧은 간격으로 결제 요청이 반복되어도 중복 처리되지 않아야 한다.
- Booking API는 같은 요청이 여러 번 들어와도 주문, 결제, 재고가 중복 반영되지 않도록 설계해야 한다.

### 4.4 결제 확장성

- 신용카드, Y 페이, Y 포인트를 지원해야 한다.
- 복합 결제는 `신용카드 + 포인트` 또는 `Y 페이 + 포인트`를 허용한다.
- 신용카드와 Y 페이는 함께 사용할 수 없다.
- 향후 새로운 결제 수단이 추가되어도 Booking API의 비즈니스 로직 변경을 최소화해야 한다.
- 결제 확장 구조와 선택 근거를 `DECISIONS.md`에 기록해야 한다.

### 4.5 장애 대응 및 예외 처리

- Redis 장애 시 fallback 전략을 수립하고 근거를 제시해야 한다.
- 한도 초과 등 결제 실패 케이스에 대한 대응 로직을 설계해야 한다.
- 장애 대응과 예외 처리 전략을 `DECISIONS.md`에 기록해야 한다.

---

## 5. 구현 범위와 가정

- 실제 PG사 연동은 생략할 수 있다.
- 단, 인터페이스 등을 통해 실제 결제 흐름으로 확장 가능한 구조를 갖춰야 한다.
- 회원 인증과 로그인 보안 처리는 범위에서 제외할 수 있다.
- 인프라 증설이 제한적인 상황을 가정한다.

## 7. 판단 근거 기록 기준

기술 선택은 결과보다 판단 과정이 드러나야 한다.

- 어떤 문제 상황이 있었는지 기록한다.
- 어떤 선택지를 검토했는지 기록한다.
- 각 선택지의 트레이드오프를 기록한다.
- 최종적으로 왜 해당 방향을 선택했는지 기록한다.
- 추가 인프라를 사용했다면 이유와 비용 대비 효과를 함께 설명한다.
